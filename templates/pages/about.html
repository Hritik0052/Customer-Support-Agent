About us

Support teams drown in triage, not in solving.

SupportIQ started from a simple observation: agents spend more time reading, labelling and routing tickets than actually fixing problems. That work is real, repetitive, and exactly what a language model is good at.

{# ---------- Story ---------- #}

The problem

A support agent opening a fresh queue has to answer the same questions for every single ticket. What is this about? How urgent is it? Is this person angry? What do I even say back?

Answering those four questions takes two to three minutes per ticket. On a hundred-ticket day that is four hours of reading before a single problem gets solved. Worse, it is inconsistent: the same ticket gets labelled differently depending on who opens it and how tired they are.

Rules-based routing was the old answer, and it breaks constantly. Keyword rules cannot tell the difference between "I was charged twice" and "I want to be charged annually instead".

The approach

SupportIQ sends each ticket through a single, carefully constrained prompt and gets structured JSON back: category, priority, sentiment, a summary, a confidence score, and a suggested reply.

The interesting engineering is not the API call — it is everything around it. Language models are enthusiastic and imprecise. They wrap JSON in markdown fences, invent category names that were never offered, and return confidence as "85%" when asked for a number between zero and one.

So the parser assumes the model will misbehave and copes: it strips fences, finds the first brace-balanced object, repairs trailing commas, and maps invented categories back onto the real ones. A bad response degrades into a sensible default instead of a stack trace.

{# ---------- Principles ---------- #}

What we optimised for

Fails softly

When the AI is unavailable or returns nonsense, the ticket is still saved and you get a retry button. An outage never loses a customer's message.

Your data stays yours

Tickets live in your own database. Each account only ever sees its own tickets, enforced at the query layer rather than hidden in the UI.

Open all the way down

Open models through OpenRouter, an open framework, and an AI layer you can point at a different provider by editing one file.
